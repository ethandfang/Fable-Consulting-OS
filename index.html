Internal Reference · Claude Fable 5 for Consulting

Twenty-one skills.
One operating system.

A working reference for running a repeatable six-stage consulting method on Claude Fable 5 through Claude Code. Each skill is a file you install once and invoke as a command. This document covers what each skill produces, when to reach for it, and the exact setup — assuming you've used ChatGPT but not Claude Code.

July 2026 v2 — Working reference Internal — do not distribute externally

Executive summary

The system in thirty seconds

Every engagement runs the same six stages in order: Diagnose → Map → Choose → Execute → Govern → Communicate. Each stage has 3–4 skills — packaged methods with a fixed output format — installed as files in Claude Code and invoked either automatically or by typing /skill-name. Stage six feeds back into stage one: the board's decision becomes the next engagement's starting condition.

00

Setup

Getting started

Three concepts carry everything in this document: skills (procedures Claude loads on demand), CLAUDE.md (facts Claude loads every session), and MCP (connections to live data). If you know these, the remaining sections are just content.

Skills

A folder containing a SKILL.md file: YAML frontmatter that tells Claude when to use it, then markdown instructions. The folder name becomes the command — .claude/skills/decision-memo/SKILL.md creates /decision-memo. Claude also invokes skills on its own when your request matches the description. Slash commands and skills are the same mechanism (docs).

CLAUDE.md

A markdown file at the project root, loaded into every session automatically. Use it for standing facts: client name, engagement scope, terminology, what's out of bounds. Procedures don't belong here — a skill's body loads only when used, so long method material costs nothing until you need it (docs).

MCP

The Model Context Protocol connects Claude Code to external systems — databases, drives, issue trackers — so it reads live data instead of what you paste. Servers are added with claude mcp add and authenticated with the /mcp command (docs).

Prerequisites

You need a terminal, a Claude subscription or API access, and Claude Code installed:

# macOS / Linux (native installer)
curl -fsSL https://claude.ai/install.sh | bash

# or via npm
npm install -g @anthropic-ai/claude-code

Then run claude in any folder to start a session. Run /model inside the session and select Fable 5 — the analysis-heavy skills in stages 02–05 benefit from its long-horizon reasoning; routine drafting works fine on Opus or Sonnet.

Five-minute quickstart

  1. Create one skill folder

    Personal skills live in ~/.claude/skills/ and follow you across projects. Run: mkdir -p ~/.claude/skills/assumption-audit

  2. Write the SKILL.md

    Save the file below to ~/.claude/skills/assumption-audit/SKILL.md. The description is how Claude decides when to load it unprompted; the body is the method it follows.

  3. Invoke it

    In a session, type /assumption-audit followed by a strategy statement — or just describe a strategy and ask what has to be true for it to work; Claude loads the skill on its own. Edits to the file take effect live, no restart needed.

---
name: assumption-audit
description: Audit the implicit assumptions behind a strategy or plan.
  Use when the user presents a strategy, plan, or business case and asks
  what could go wrong, what it depends on, or whether it holds up.
---

Audit the assumptions behind: $ARGUMENTS

1. List every assumption the strategy silently makes — market, customer,
   competitor, execution, and financing assumptions separately.
2. For each: rate the kill-risk (would the strategy fail if this is
   wrong?) and confidence (what evidence do we actually have?).
3. Output a table sorted by kill-risk, then name the three cheapest
   tests that would retire the most risk.
Do not soften findings. If an assumption is untested, say so.

The $ARGUMENTS placeholder receives whatever you type after the command. That's the entire mechanism — the remaining twenty skills are the same pattern with different methods.

Where skills live — and how the team gets them

~/.claude/skills/ is personal (all your projects). A project's .claude/skills/ applies to that project only — and it's a directory you commit to git, which is how the full set reaches other consultants (section 07). Two frontmatter fields matter for control: disable-model-invocation: true means only you can trigger the skill (use for deliverable-builders you want to time), and user-invocable: false means only Claude can (use for background knowledge). Keep each SKILL.md under ~500 lines; move templates and reference material to separate files in the same folder.

01

Stage — Diagnose

Problem framing

Engagements fail in week one, not week twelve. The Diagnose skills force the client's question into a testable form before any analysis is commissioned — because the stated problem ("revenue is declining, fix marketing") is usually a symptom, and analyzing a symptom produces a well-evidenced answer to the wrong question.

All three skills here are diagnostic reads, not deliverable builders. They run early, they run cheap, and their output is the input to every later stage: the situation assessment sets the fact base, the barrier diagnosis names the constraint the engagement is actually about, and the assumption audit tells you which analyses in stage 02 are worth buying.

The skills

01Situation assessment›

Produces a one-page fact base before anyone proposes a fix: market position, margin trend, momentum versus peers, and — separately — the client's stated question versus what the numbers suggest the real question is. Ends with the three facts that most constrain the answer.

02Growth barrier diagnosis›

Produces a ranked list of candidate constraints (demand, capacity, capability, capital, competition) with the evidence for and against each — separating the constraint that actually caps growth from the noise that merely surrounds it. Names the single binding constraint and what data would confirm it.

03Assumption audit›

Produces a table of everything the strategy silently assumes, each rated for kill-risk and evidence, plus the three cheapest tests that would retire the most risk. This is the skill built in the quickstart above — the pattern generalizes to the whole library.

Before / after

Without the skill

Our client's revenue has been declining for 6 quarters. Analyze why and suggest how to fix it.

Claude produces a broad, plausible essay — pricing, churn, competition, macro — with no priority order and no test of whether "declining revenue" is even the right frame.

With /situation-assessment

/situation-assessment Client is a $400M industrial distributor, revenue -4% for 6 quarters, EBITDA flat, top competitor growing 8%. Board asked us to "fix sales effectiveness."

The skill forces the format: fact base first, stated-vs-real question flagged (flat EBITDA on falling revenue suggests mix shift, not a sales problem), missing data requested by name before conclusions.

Reach for this stage at kickoff, or any time a request arrives pre-framed as a solution ("we need a pricing study"). Skip to 02 only when the problem statement has already survived a challenge — if you can't say what evidence would prove the frame wrong, you're not done here.

How to actually run it

Workflow — model-invocable personal skills

These three are plain skills with strong description fields, left model-invocable: you want Claude reaching for them uninvited whenever a conversation smells like an unframed problem. No special frontmatter needed beyond the description. Install once in ~/.claude/skills/; they then apply to every client folder you open.

One upgrade worth making: give situation-assessment live context using dynamic context injection — a !`command` line in the skill body runs before Claude reads the skill, and its output is inlined. If each engagement repo keeps a facts/ folder, the skill can pull it automatically:

## Current engagement fact base
!`cat facts/*.md 2>/dev/null || echo "No fact base found - ask for one."`

Copy-paste starting prompt:

/situation-assessment [client], [size], [industry]. Board's stated ask:
"[verbatim ask]". Key numbers: [3-5 metrics with trend]. What is the
real question, and what data do we need before answering it?
02

Stage — Map

Market & competitive intelligence

The Map skills turn "summarize the market" into structured intelligence: where value concentrates, how rivals are positioned to move, and which white space is genuinely unclaimed rather than merely unserved. Their common discipline is reconciliation — every sizing is built two ways and the gap is explained, every rival claim is tied to a source.

This is the stage where Claude's ability to pull live data matters most, which is why the workflow block below is about tools, not prompts. A market map built from training-data memory is a liability; the same skill grounded in current filings, pricing pages, and your client's own data is a defensible fact base.

The skills

04Market mapping›

Produces a market sizing built top-down (industry reports, category spend) and bottom-up (customers × frequency × price), reconciles the two, and calls out growth pockets explicitly. The reconciliation gap is reported, not hidden — a 2× gap is itself a finding.

05Competitive intelligence›

Produces a profile per rival — economics, cost position, incentives, recent moves — ending with each one's three most likely next moves and the leading indicators that would confirm them. Sourced claims only; speculation is labeled as such.

06Customer segmentation›

Produces segments cut by need and willingness to pay — not by the org chart's convenient categories (SMB/mid/enterprise). Each segment gets size, growth, current share, and the one thing that would make it switch.

07Profit pool analysis›

Produces a map of where the industry's profit (not revenue) accumulates along the value chain, where it's migrating, and which step the client occupies. The standard finding: the client is fighting for share in a step whose pool is draining.

Before / after

Without the skill

Give me an overview of the US industrial-fasteners market.

A competent Wikipedia-grade summary: size, players, trends. Nothing you couldn't have gotten in one search, and nothing about where the money is.

With /profit-pool

/profit-pool US industrial fasteners. Client sits at distribution. Search for current data - I need the value chain from raw steel to end user, margin at each step, and where margin is moving.

The skill's format forces a chain-step table with margin per step and a migration read — and because the skill grants search access, each number carries a source you can check.

Reach for this stage once Diagnose has fixed the question — Map describes the board; Choose (03) picks the move. Running Map before Diagnose produces beautifully-sourced answers to unvetted questions. If mid-analysis the fact base contradicts the frame, go back to 01; that's the loop working, not a failure.

How to actually run it

Workflow — skills with search access + MCP data sources

Two setup moves distinguish this stage. First, grant the skills web access in frontmatter so research doesn't stall on permission prompts — allowed-tools pre-approves listed tools while the skill is active:

---
name: competitive-intel
description: Build a sourced competitive profile with predicted moves.
  Use for questions about competitors, rivals, or market positioning.
allowed-tools: WebSearch WebFetch
---
For each competitor in $ARGUMENTS: search for current financials,
recent strategic moves, and stated priorities. Profile economics and
incentives, then predict the three most likely next moves with the
leading indicator for each. Every factual claim carries a source link.

Second, connect client data through MCP so Claude reads it directly instead of you pasting exports. Servers are added once per engagement and shared with the team via the project's .mcp.json (use --scope project):

# Remote server over HTTP (most hosted connectors)
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Local server, e.g. a Postgres bridge to client data
claude mcp add --scope project --env DATABASE_URL=$CLIENT_DB pg \
  -- npx -y @modelcontextprotocol/server-postgres

# Authenticate OAuth servers from inside the session
/mcp

Browse vetted connectors in the Anthropic directory. Treat MCP servers like any vendor: verify before connecting, especially for engagements under NDA.

03

Stage — Choose

Strategic choice

The Choose skills exist to prevent the most common failure of AI-assisted strategy work: a single confident recommendation with no visible alternatives. Each one compares paths on economics, sensitivities, and risk before recommending — and the rejected options stay in the output, with the reason they lost.

These are the most computation-heavy skills in the library. A business case is only as good as its sensitivity analysis, and a portfolio review is only as good as its scoring discipline — which is why the workflow below pins these skills to high effort and ships them with templates, so the model spends its reasoning on the numbers rather than reinventing the format.

The skills

08Strategic options›

Produces three to five genuinely distinct paths — not one plan at three price points — each with its core bet, rough economics, the conditions under which it wins, and the earliest signal it's failing. Ends with a recommendation and the explicit reason each alternative lost.

09Pricing strategy›

Produces a price architecture built from willingness-to-pay by segment (stage 02's cut) and expected competitive response — not cost-plus habit. Includes the migration path from current prices and the volume-at-risk math for each step.

10Business case›

Produces the full case — NPV, payback, IRR — with the three sensitivities that actually swing the answer stated up front, not buried in an appendix. Every input assumption is listed with its source, so the case can be audited line by line.

11Portfolio review›

Produces a scoring of every business unit on market attractiveness and right to win, with explicit invest / hold / exit calls and the capital reallocation they imply. Forced ranking — no unit is allowed to be "strategic" without a number behind it.

Before / after

Without the skill

Should our client expand into Mexico or double down on the US market? Build a business case.

One spreadsheet-shaped answer, usually favoring whichever option the prompt named first, with sensitivities as an afterthought.

With /business-case

/business-case Mexico entry vs US densification for [client]. Inputs in facts/case-inputs.md. Use 12% WACC, 5-yr horizon. I want both cases in the same template, the three swing sensitivities named up front, and the break-even value on each.

The template forces identical structure for both options, so the comparison is real. Sensitivities lead the output; the recommendation follows the math instead of preceding it.

Reach for this stage when the fact base (02) is stable enough that the options can be priced. Not before — an options analysis on a shaky fact base just launders uncertainty into confident-looking NPVs. If two options are within noise of each other, that's a finding: the decision hinges on a belief, and stage 05's war-gaming should test it.

How to actually run it

Workflow — templated skills pinned to high effort

Two frontmatter features carry this stage. effort overrides the session's reasoning-depth setting while the skill is active — pin the quantitative skills to xhigh so sensitivity analysis gets full reasoning even when the rest of the session runs lighter. And supporting files keep the SKILL.md short: put the case template in the skill folder and reference it, so the format loads only when the skill runs.

---
name: business-case
description: Build a full investment case with NPV, payback, and
  named swing sensitivities. Use for invest/divest/enter decisions.
effort: xhigh
argument-hint: [decision] [key inputs or file]
---

Build the case for: $ARGUMENTS

Follow the structure in [template.md](template.md) exactly.
Rules: state the 3 swing sensitivities before the recommendation;
list every input assumption with its source; show the break-even
value of each swing variable; flag any input you had to invent.

The folder is then business-case/SKILL.md + business-case/template.md. Copy-paste starting prompt:

/strategic-options [client] needs [outcome] within [horizon].
Constraints: [capital, capability, appetite]. Generate 3-5 distinct
paths - different bets, not variants of one bet. For each: economics,
win conditions, earliest failure signal. Then recommend and show why
the others lost.
04

Stage — Execute

Execution

The Execute skills convert a chosen direction into things a client organization can actually run: named owners, decision rights, a sequenced initiative list, and a first 90 days specific enough to start Monday. The test for every output here is assignability — if no row of the deliverable maps to a person and a date, it's still strategy, not a plan.

Unlike stages 01–03, these skills are configured so only you can trigger them. A roadmap generated because the conversation drifted near the topic is noise; a roadmap generated when the prioritization is settled is the deliverable. The frontmatter flag that enforces this is the main workflow point below.

The skills

12Operating model design›

Produces structure, decision rights, and capability requirements derived from the chosen strategy — not inherited from the last reorg. Output is a decision-rights table (who decides, who's consulted, who executes) per critical decision type, plus the capability gaps and how each gets closed: build, buy, or borrow.

13Initiative prioritization›

Produces the initiative long-list forced through a value-versus-feasibility screen until only a fundable sequence remains. Every cut initiative is listed with its score — so when a sponsor asks why their project died, the answer is a number, not a judgment call.

14Transformation roadmap›

Produces phases with owners, dependencies, and stage gates — each gate defined as an observable condition, not a date. Opens with a first-90-days section granular enough that week one has named actions.

Before / after

Without the skill

Create an implementation roadmap for the strategy we discussed.

Four phases named "Foundation, Build, Scale, Optimize" with no owners, gates measured in quarters, and a Gantt chart nobody will open twice.

With /transformation-roadmap

/transformation-roadmap Initiatives and scores are in initiatives.csv (already prioritized). Org constraints: no new hires before Q1, CFO owns any initiative touching pricing. Phase by dependency, define each gate as an observable condition, and make the first 90 days week-by-week.

The skill inhales the live CSV via dynamic context injection, respects the stated constraints as hard rules, and the 90-day section names owners because the skill's format demands a name per row.

Reach for this stage when stage 03's choice is made and defended — not merely drafted. If trade-offs are still open, a roadmap just hides the unresolved decision inside phase two. Operating model before roadmap: sequencing initiatives into an org that can't own them produces a plan that's failed on arrival.

How to actually run it

Workflow — user-triggered skills that pull live working files

Set disable-model-invocation: true on all three. This removes the skill from Claude's automatic-trigger pool — it runs only when you type the command, which is what you want for deliverable-builders whose timing matters. Combine with dynamic context injection so the skill reads the current state of the engagement's working files at the moment you invoke it:

---
name: transformation-roadmap
description: Build a phased roadmap with owners, gates, and a
  week-by-week first 90 days from a prioritized initiative list.
disable-model-invocation: true
argument-hint: [constraints]
---

## Current prioritized initiative list
!`cat initiatives.csv 2>/dev/null || echo "initiatives.csv not found"`

## Constraints from the consultant
$ARGUMENTS

Build the roadmap: phases sequenced by dependency, one named owner
per initiative, each stage gate an observable condition. Open with
a week-by-week first 90 days. Flag any initiative whose owner or
dependency is missing from the list rather than inventing one.

The !`cat initiatives.csv` line runs before Claude reads anything, so the roadmap is always built from the file as it exists right now — edit the CSV, re-run the command, get a consistent regeneration.

05

Stage — Govern

Risk & value governance

Most strategies aren't wrong — they're unguarded. The Govern skills pressure-test the plan before the market does and instrument it so value capture is visible in quarters, not discovered missing in years. The output standard: every risk has an owner, a trigger, and a pre-agreed response; every board metric traces to a driver a team can move.

War-gaming is the technically interesting one. Playing a competitor honestly requires not knowing your side's private reasoning — which maps exactly onto a Claude Code feature: forked subagent contexts that start clean, without your conversation history. The red team genuinely can't peek.

The skills

15War-gaming›

Produces each counterparty's best response to your strategy — competitors, regulators, key customers, each played in turn from its own incentives — surfacing the counter-moves the plan hasn't priced in. Output is a move/counter-move tree with the two responses most damaging to the case.

16Risk mitigation›

Produces a risk register where every material risk carries an owner, an observable trigger, and a pre-agreed response — not a heat map that dies in an appendix. Risks without a plausible trigger are challenged as either not material or not yet understood.

17KPI architecture›

Produces a metric tree linking each board-level outcome down through drivers to the operational metrics teams actually control — with the math between levels explicit, so a miss at the top is traceable to a driver in minutes.

18Value realization›

Produces a scorecard tracking captured value against the stage-03 business case, initiative by initiative, so drift shows up in quarters. Includes the leading indicators that predict a miss before the financials show it.

Before / after

Without the skill

What are the risks to this strategy and how would competitors respond?

A generic risk list and a competitor response written by the same context that built the strategy — so it politely attacks the strategy's weak points the strategy already defends.

With /war-game

/war-game strategy-summary.md Counterparties: [Competitor A], [Competitor B], [state regulator]. Play each from its own incentives. I want the two most damaging responses and what our pre-committed counter should be.

Each counterparty runs in a fresh forked context that has only the public strategy summary — no access to your private reasoning — so the attacks land where you didn't think to defend.

Reach for this stage after 03 produces a recommendation and before 06 communicates it — a memo that hasn't been war-gamed gets war-gamed live by the board instead. KPI architecture and value realization belong at the start of execution (alongside 04), not after: a scorecard designed post-launch measures what happened to be trackable, not what matters.

How to actually run it

Workflow — forked subagent contexts for honest red-teaming

Setting context: fork in frontmatter runs a skill in an isolated subagent: it gets the SKILL.md content as its entire prompt and none of your conversation history. That isolation is the point — the war-game skill passes each counterparty only the public strategy document, so the red team argues from what the market can see:

---
name: war-game
description: Play counterparties against a strategy from their own
  incentives, in an isolated context with no access to our reasoning.
context: fork
disable-model-invocation: true
argument-hint: [strategy file] [counterparties]
---

You are a war-game engine. Input: $ARGUMENTS

For each named counterparty, in turn:
1. State its objectives, constraints, and incentives from public facts.
2. Generate its strongest response to the strategy - the move it would
   actually make, not the move that is convenient for our client.
3. Score the damage to the strategy's business case (which you cannot
   see - infer the exposure from the strategy itself).
Conclude: the two most damaging responses and the pre-commitment that
would blunt each.

The remaining three skills in this stage are ordinary inline skills — risk-mitigation and kpi-architecture want your full engagement context, so don't fork them. Starting prompt for the scorecard:

/value-realization Track against the case in business-case-v3.md.
Initiative-level: planned vs captured value, leading indicator per
initiative, and the earliest date a miss becomes visible. Output as
a table the PMO can update monthly.
06

Stage — Communicate

Executive communication

Analysis that can't survive a board meeting didn't happen. The Communicate skills convert the preceding five stages into a Pyramid-Principle storyline — answer first, arguments beneath it, evidence at the base — plus the two artifacts around it: a stakeholder plan for getting the decision made, and a one-page memo that makes it official.

The hostile-Q&A pass is where these skills earn their keep. Claude drafts the memo, then a separate forked context argues against it as the most skeptical board member — same isolation mechanism as war-gaming, aimed at your own deliverable. Weak joints surface in rehearsal instead of in the room.

The skills

19Stakeholder alignment›

Produces a map of who wins, who loses, and who can block — with a sequenced plan for the conversations before the meeting. The standard rule enforced: no decision-maker hears the recommendation for the first time in the room where they must approve it.

20Narrative builder›

Produces the storyline before the slides: governing thought at the top, three to four supporting arguments, evidence mapped beneath each. Output is a slide-by-slide skeleton where every header is a claim, not a topic — "Mexico entry returns 2.1× US densification," not "Market Analysis."

21Decision memo›

Produces one page: the recommendation, the cost of waiting, the risks being knowingly accepted, and the signature line. If it doesn't fit on a page, the skill says the thinking isn't done and names what's still unresolved rather than shrinking the font.

Before / after

Without the skill

Write an executive summary of our analysis for the board deck.

A faithful chronological recap of the work — methodology first, recommendation on the last slide, exactly backwards for an executive audience.

With /narrative + hostile Q&A

/narrative-builder Full analysis in findings/. Governing thought: exit the retail segment within 18 months. Audience: PE board, CFO is skeptical. then: /hostile-qa Attack the narrative above as the skeptical CFO. Ten hardest questions, plus which ones we cannot currently answer.

Answer-first structure by construction, then a clean-context adversary that flags the two questions with no good answer — a week before the CFO finds them.

Reach for this stage last — but reach for stakeholder alignment early, in parallel with 03: the map of who blocks changes which option is even feasible, and learning that after the memo is drafted means redoing 03. The memo itself is always the final artifact; it freezes the decision the other twenty skills built.

How to actually run it

Workflow — template-backed skills, with arguments for the audience

These skills live or die on format discipline, so each ships with its template as a supporting file (the stage-03 pattern). The useful extra here is named arguments — declared in frontmatter, they slot into the body by name, which keeps audience and decision separate instead of mashed into one string:

---
name: decision-memo
description: Produce a one-page board decision memo.
disable-model-invocation: true
arguments: [decision, audience]
argument-hint: [decision] [audience]
---

Write a one-page decision memo for $audience recommending $decision.
Structure per [template.md](template.md): recommendation (2 sentences),
cost of waiting (quantified), risks knowingly accepted (from the
stage-05 register), decision requested, signature line.
Hard rule: one page. If it does not fit, the thinking is not done -
say what is unresolved instead of compressing.

Invoked as /decision-memo "exit retail segment" "PE board" — the quoted strings map to $decision and $audience in order. Since these skills produce client-facing text, keep them user-triggered (disable-model-invocation: true): you decide when a memo exists.

07

The operating system

Assembling the system

The twenty-one skills aren't a menu — they're stations on a loop, and the loop only works if the whole team runs the same one. Three mechanics turn the skill files into an operating system: a shared engagement repo, a CLAUDE.md that carries the client context, and per-engagement data connections.

Communicate closes back into Diagnose: the board's decision becomes the next engagement's starting condition.

The engagement repo

One git repository per engagement, with this shape:

acme-engagement/
├── CLAUDE.md              # client facts, scope, terms, boundaries
├── .claude/
│   └── skills/            # all 21 skills - committed, versioned
│       ├── situation-assessment/SKILL.md
│       ├── business-case/
│       │   ├── SKILL.md
│       │   └── template.md
│       └── ...
├── .mcp.json              # shared data connections (project scope)
├── facts/                 # the evolving fact base (stage 01-02)
├── findings/              # analysis outputs (stage 02-05)
└── initiatives.csv        # working files the skills read live

A new team member runs git clone then claude, accepts the workspace trust prompt, and has the identical system: same skills, same templates, same data connections. That trust prompt matters — project skills and .mcp.json servers activate only after you approve the workspace, which is also your review checkpoint for repos you didn't author. Keep firm-level method skills in ~/.claude/skills/ (or package them as a plugin for firm-wide distribution); keep client-tuned variants in the repo — where names collide, the more specific level wins.

CLAUDE.md carries the context between skills

Skills are stateless procedures; CLAUDE.md is the standing memory every one of them reads. Keep it to facts that change rarely and matter always:

# ACME Corp — Retail Exit Engagement

## Client
$400M industrial distributor, PE-owned (Meridian Capital, 2023).
Fiscal year ends March. All figures USD unless noted.

## Engagement scope
Question: exit vs fix the retail segment. Decision meeting: Sept 12.
Out of scope: management changes, anything touching the union contract.

## Conventions
- WACC 12% for all cases (agreed with CFO, 2026-06-14)
- "Retail" = segments 40-49 in the GL, not the storefront business
- Sensitive: do not cite the Meridian LP letter in client-facing docs

The conventions section is what keeps twenty-one skills consistent — every business case uses the same WACC, every memo respects the same boundaries, without repeating it in every prompt.

Running a stage end-to-end

A typical stage-03 working session, showing how the pieces chain — each skill's output lands in the repo, and the next skill reads it from there:

# in the engagement repo
claude

# 1. generate the option set from the stage-02 fact base
/strategic-options Exit vs fix retail. Fact base in facts/,
constraints in CLAUDE.md. Write output to findings/options-v1.md

# 2. price the two leading options identically
/business-case Options 1 and 3 from findings/options-v1.md,
same template, inputs from facts/case-inputs.md

# 3. pressure-test before it goes anywhere near a slide

Maintaining the system

Treat skills like code. When a skill produces a weak output, fix the SKILL.md and commit — the edit takes effect live in-session, and the next consultant inherits the fix. To test changes systematically, the skill-creator plugin (/plugin install skill-creator@claude-plugins-official) runs with-skill versus without-skill comparisons and grades outputs, so a template change is measured rather than vibes-checked. Version the templates hardest — they're the firm's accumulated judgment about what a deliverable looks like, and they compound.
